# T4 入札参加資格申請書＋メール作成アプリ 設計書 v1

作成：サトル（要件定義パートナー）／2026-07-08
対象：山元光樹社長（テマヒマ・ラボ）

## 0. 前提・分析方法

`logs/T4_入札サンプル/構造抽出.md`（5サンプルの全シート構造）に加え、**5つの実ファイルすべてを Python
（xlrd / openpyxl）で直接開き、セルの色・条件付き書式・結合セル・図形（drawing）の有無を実際に検証した**。
想像では設計していない。確認できた事実は次のとおり。

| チェック項目 | 確認結果 |
|---|---|
| セルの背景色（塗りつぶし） | サンプル①〜⑤の全シートで、フィールド入力欄に静的な背景色は**一切ない**（真っ白のグリッド） |
| 条件付き書式（`conditionalFormatting`タグ） | 5サンプルとも**0件**。これらは"提出前の未記入の官公庁公式様式"なので、そもそも社長が言う「未入力＝色つき」の仕組みは元のExcel（省庁配布ファイル）には無い |
| 結合セル | サンプル④の主要シートだけで50件以上。ラベル部分（タイトル・見出し）は結合、**1文字入力エリア（フリガナ等）は結合なし＝1セル1文字が徹底**されている |
| 列幅 | サンプル④で列幅 `0.90625`（約半角1文字分）が最大200列以上連続。これが「1文字1セル」グリッドの正体 |
| 図形（drawing.xml） | サンプル④・⑤で確認。中身は**すべて直線（`prst="line"`）と説明用の四角形テキストボックス1つ**。○（oval）や✓のような図形は見つからなかった |
| 「○で囲む」指示 | サンプル①業態調書に `有 / 無 （いずれかに○）` という**手書き前提の選択欄**を発見（row8〜9）。丸を描く場所は決まっているが、Excel上に丸graphicは元から無い＝**元々は紙時代の名残で、Excel化するときに毎回"自己流"で表現方法（文字○／セル色／図形）を作らざるを得ない箇所**と判明 |

→ **重要な発見**：社長が言う「条件付き書式が壊れた」「○の図形とセルがズレた」という過去の失敗は、
**省庁配布の元Excel自体の機能ではなく、過去にスプレッドシート化する際に"社長側で独自に追加した"仕組み**
だったと考えられる。つまり今回は「壊れた機能を復元する」のではなく、**同じ轍を踏まないやり方で新規に設計する**のが正しい。

## 2. アプローチの提案：「原本ロック＋入力フォーム分離＋自動転記」の3層構成

過去の失敗（条件付き書式の崩壊／図形とセルのズレ）は、どちらも**"250列の巨大な文字グリッドに人間が直接触る"**
ことが根本原因。これを構造的に断つため、ゼロから作り直さず、**社長の持つ実物レイアウトをそのまま活かしながら
「人が触る場所」と「機械が触る場所」を完全に分離する**方式を提案する。

```
┌─────────────────┐     transcribe      ┌──────────────────────────┐
│ ① 入力フォームシート │ ───(GAS自動転記)──▶ │ ② マスターシート群（省庁別） │
│  ・1フィールド=1行    │                     │  ・元Excelのレイアウトを     │
│  ・人間はここだけ入力  │                     │    セル単位でそのまま複製     │
│  ・ISBLANK系CFで      │                     │  ・保護（人間は編集不可）     │
│    未入力を黄色ハイライト│                     │  ・GASのみが書き込む         │
└─────────────────┘                     └──────────────────────────┘
                                                        │ 印刷/PDF化してそのまま提出
                                                        ▼
                                              ③ メール文章生成（GAS）
                                              件名・本文を自動組み立て
```

- **① 入力フォームシート**：フィールドごとに1行（会社名、フリガナ、郵便番号、住所、代表者役職・氏名、
  建設業許可番号、資本金、有資格技術職員数……）の普通サイズのシート。社長はここだけ触る。
  250列グリッドを見る必要は一切ない。
- **② マスターシート群**：省庁配布Excelのシートをそのまま複製・保護したもの。これは「印刷して提出する原本」
  としての役割に徹し、**人間による直接編集を禁止**（シート保護＋範囲保護をかける）。書き込むのはGASのみ。
- **③ メール文章生成**：①の入力内容を使って提出メールの下書きを作る（詳細は5章）。

この方式が過去の失敗を構造的に防げる理由：

| 過去の失敗 | この設計での防止策 |
|---|---|
| 条件付き書式が壊れた（250列グリッドに複雑なCFをかけて崩壊） | CFは①の**普通サイズの入力フォーム**（40〜60行程度）にだけかける。250列グリッド側（②）にはCFを一切置かない |
| ○を図形で置くべきところをセルに文字入力してズレた | ②のマスターシートは**人間が触れない**（保護済み）。○に相当する「有／無」選択は①でプルダウン選択→GASが②の決まった座標に自動反映。人間が座標を数えて図形やセルに手動入力する工程自体をなくす |

## 3. 条件付き書式の正しい再現方法（未入力＝色つき、入力で色が消える）

Google スプレッドシートの条件付き書式は、**「条件を満たす間だけ書式を適用し、満たさなくなれば自動で書式を外す」**
という仕様のため、本来はExcelより素直に動く。崩れた原因は十中八九「①巨大結合範囲に1本のルールを当てて範囲がズレた」
「②"入力済みで色が消える"と"未入力で色がつく"を別々の2ルールで作って競合した」のどちらか。

今回の設計での作り方：

1. **入力フォームシート①のB列（入力セル）ごとに、1セル＝1ルールで設定する**（複数セルへの一括適用は`sqref`崩れの元なので避け、
   同じ書式のルールを"個別に複数件"作るか、"1列全体"のように**単純な矩形範囲のみ**に絞る）。
2. ルールは**1本だけ**：
   - 条件：「カスタム数式」
   - 数式：`=ISBLANK($B2)`（対象セルが空欄なら真）
   - 書式：背景色＝黄色
   - → 入力されればGoogle側が自動でルールがfalseになり、勝手に色が消える。**「消える側」の逆ルールを別途作らない**（これが過去の競合の原因になりやすい）。
3. **必須項目だけ**に絞る（全フィールドに付けると見づらくなり、社長が「結局どれが本当に必須か」を見失う）。
   必須：商号又は名称／本社住所／代表者役職氏名／建設業許可番号／法人番号／希望工種区分。
4. シート先頭に「進捗サマリ」セルを置き、`=COUNTIF(B2:B60,"")&"件 未入力"` のような集計式で
   **一目で残作業件数がわかる**ようにする（250列を目でスクロールして探させない）。
5. マスターシート側（②）には**CFを置かない**。転記が終わった時点で全項目は「値が入っている」状態になるので、
   未入力チェックは①だけで完結させる。

## 4. 図形 vs セル入力の切り分けルール

社長が二度と迷わないよう、**全フィールドを3色に分類したガイドシート「入力ガイド」を①の一番手前に置く**。

| 色 | 分類 | 対応するフィールド例 | 触ってよい場所 |
|---|---|---|---|
| 🟩 緑：直接入力 | テキスト・数値をそのまま書く | 商号、住所、資本金、完成工事高、電話番号 | ①入力フォームにテキストで入力 |
| 🟨 黄：選択式（プルダウン） | 二択・多択で「印をつける」ことになっている項目 | 新規／更新、有／無（コンプライアンス審査、退職者再就職）、希望工種区分 | ①入力フォームの**プルダウン**（データの入力規則）で選ぶだけ。○を書く操作は一切させない |
| 🟥 赤：完全ロック（自動転記専用） | ②マスターシート本体・下線や罫線などの装飾図形 | 250列の文字グリッド全体、直線の下線図形（フリガナ欄の罫線など） | **人間は触らない**。GASが①→②へ自動で書き込む |

黄色（選択式）の項目は、従来「セルに○を手書き」していたためにズレの原因になっていた箇所。
今回は**「①でプルダウン選択→GASが②の該当セルに"○"の1文字だけを、決まった座標にプログラムで書き込む」**
方式に統一する（座標は構造抽出.mdで既に特定済みなので、人間が数える必要はない）。
どうしても視覚的に丸を掛けたい場合も、GASの`Range.insertCheckbox()`や固定位置への図形挿入をコードで行い、
**手動ドラッグでの図形配置は禁止**する（これが位置ズレの再発を防ぐ最大のポイント）。

## 5. 共通化の設計：「1ファイル＋省庁別タブ」構成を推奨

**5サンプルを比較すると、シートの意味・フィールドの8割は共通、差分は宛先文言・シート枚数・工種コードの並びの3点に集約される。**

- 共通：申請書1枚目の基本情報欄の構成、業態調書の資本関係・退職者再就職状況の様式、共同企業体調書の技術職員数マトリクス
- 差分：①宛先（省庁名・部署名・年度表記）②シート枚数（③は港湾関連で12枚、⑤は3枚のみ）③工種コードの表記ゆれ（「ほ装」/「舗装」など）

### 推奨構成：**1ファイル内で省庁ごとにシートを分ける**（省庁ごとに別ファイルにはしない）

理由：
- 別ファイルにすると、入力フォーム・転記スクリプト・メール生成機能を**省庁の数だけ二重管理**することになり、
  「めんどくさいを減らす」という会社の方針に反する。
- 1ファイルに集約すれば、**入力フォームは1つのまま**、GASの転記先を「プルダウンで選んだ省庁」に応じて
  切り替えるだけで済む。新しい省庁を追加するときも、その省庁のExcelをシートとしてインポートし、
  対応表（設定シート）に1行追記するだけで拡張できる。

```
【1つのGoogleスプレッドシート】
 ├─ シート「入力フォーム」        … 共通・1枚だけ（社長がここだけ触る）
 ├─ シート「入力ガイド」          … 3色の凡例（4章）
 ├─ シート「設定」                … 省庁プルダウンの選択肢／フィールド座標マッピング
 ├─ シート「[文科省]様式1-1」     … 原本レイアウトをそのまま複製・保護
 ├─ シート「[文科省]様式1-2」
 ├─ シート「[文科省]様式2」
 ├─ シート「[防衛省]申請書_1枚目」 … 省庁が増えるごとにこの単位で追加
 ├─ シート「[防衛省]申請書_2枚目」
 ├─ …
 └─ シート「メール下書き」        … 6章
```

## 6. メール文章作成機能の設計

`office/company.js` の資料棚にある既存リンク「入札申請書＋メール作成アプリ」（GASウェブアプリ）は、
今回サトルの権限ではソースコードを確認できなかった（URLへの直接アクセスは403で拒否された）。
**ゼロから作り直すのではなく、既存アプリの仕様を踏襲・拡張する前提で設計するが、実物のApps Scriptソースの
共有を8章の確認事項として社長にお願いしたい。**

設計としては、①入力フォームの値をそのまま使い回して、以下を自動生成する：

- **件名テンプレート**：`【{提出年度}年度】競争参加資格審査申請書の提出について（{商号又は名称}）`
- **本文テンプレート**：
  ```
  {宛先省庁名} {部局名} 御中

  お世話になっております。{商号又は名称}の{代表者役職}、{代表者氏名}でございます。

  {提出年度}年度における一般競争（指名競争）参加資格審査申請書（建設工事）を
  添付のとおり提出いたします。ご査収のほど、よろしくお願いいたします。

  ---
  商号又は名称：{商号又は名称}
  本社所在地：{本社住所}
  法人番号：{法人番号}
  提出日：{提出日}
  ---
  ```
- 生成した文面は「メール下書き」シートに出力するだけで、**実際の送信は行わない**（設計図の承認フロー §6
  に準拠。外に出す操作は必ず社長自身の手で行う）。
- 添付ファイルは②マスターシート群をPDF出力したものを想定（GASの`Sheet.exportAsPdf`的な処理、または
  手動で「ファイル→ダウンロード→PDF」）。

## 7. 実装ステップ（ハック向け・段階導入）

一度に全省庁を作らず、**最小構成から型を固めて横展開**する。

| ステップ | 内容 | 目安 |
|---|---|---|
| Step 1 | サンプル⑤（文部科学省・3シートのみ＝最小構成）を対象に、①入力フォーム・②マスターシート・③転記GASのプロトタイプを作る | 型を1つ固める |
| Step 2 | 転記スクリプト（文字分解→座標への書き込み）を1フィールド（フリガナ等）で動作確認 | ロジックの正しさを検証 |
| Step 3 | 未入力ハイライト（3章のCF）を入力フォームに実装し、動作確認 | 過去の失敗①の再発防止を確認 |
| Step 4 | 「有／無」プルダウン→マスターへの自動反映を実装（過去の失敗②の再発防止を確認） | 図形問題の解決を確認 |
| Step 5 | 文科省版が安定したら、②防衛省・③農水省を横展開（設定シートにマッピング追加のみ） | 共通化の効果を検証 |
| Step 6 | 最大規模の④地方整備局（港湾含む12シート）・⑤汎用版（宛先空欄）を追加 | 最も複雑なケースへ対応 |
| Step 7 | 6章のメール文章生成機能を実装・結合 | |
| Step 8 | 実際の申請データで1回リハーサル運用（印刷・PDF化してレイアウト崩れがないか目視確認） → 本番運用開始 | |

## 8. 社長に確認したい論点

1. **既存のGAS「入札申請書＋メール作成アプリ」**（`office/company.js`のリンク集にあるURL）は、
   今回作り直す対象でしょうか、それともメール文面のテンプレート部分だけ活かして拡張しますか？
   Apps Scriptのソース（スクリプトID、または`clasp`でのコード）を共有いただけると、重複実装を避けられます。
2. 今後実際に応募する省庁は、5サンプルのうちどれが中心になりそうですか？（全省庁を最初から作るのは非効率なので、
   Step 5〜6の優先順位を決めたいです）
3. 「有・無（いずれかに○）」等の選択項目は、**紙に印刷して提出する前提**でしょうか。PDFのまま電子提出できるなら
   セルの色や太字での表現で十分ですが、「実際に丸で囲んだ見た目」が必須なら図形オーバーレイの自動生成が必要になり、
   工数が変わります。
